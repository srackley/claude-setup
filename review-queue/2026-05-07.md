**[BLOCKING] `uuid/v4` subpath import тАФ 12 source files + 4 test files**

The `uuid/v4` deep-subpath export was removed in uuid v7. All production source files except `wanderu-analytics/src/utils.ts` still use:

```js

```

With uuid v14, every such import throws `Cannot find module 'uuid/v4'` at Webpack build time and Jest module resolution. Affected packages: `wanderu-app-booking`, `wanderu-app-explorer` (├Ч2), `wanderu-ducks-live-check` (├Ч2), `wanderu-app-tripsummary`, `wanderu-components`, `wanderu-app-seo` (├Ч2), `wanderu-ducks-search`, `wanderu-app-search`, `wanderu-ducks-split-test`. Additionally, 4 test files use `jest.mock('uuid/v4', ...)`.

No compatibility shim exists тАФ confirmed by checking `jest.config.js` moduleNameMapper, Webpack `resolve.alias` across all configs, `__mocks__/` directory, and Babel transform config.

Required fix: Update all 12 source files to `import { v4 as uuidV4 } from 'uuid'` and all 4 jest mocks to `jest.mock('uuid', () => ({ v4: jest.fn(...) }))`.

### Additional Context

**CVE doesn't apply:** GHSA-w5hq-g745-h8pq is an OOB write in `v3()`, `v5()`, and `v6()` when an invalid `offset` arg is passed with a caller-supplied buffer. This codebase exclusively calls `uuidV4()` with no custom buffer/offset args тАФ the vulnerable code path is never exercised. No urgency to merge on security grounds.

**ESM/Node 22 is fine:** uuid v12+ is ESM-only, but Node 22.12+ natively supports `require()` of synchronous ESM without flags. This project runs Node 22.22.2. Not a blocker.
