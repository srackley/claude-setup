## Findings

### BLOCKING (1)

#### [1] `packages/wanderu-ducks-search/src/appstate/search/actions.js:138–141`
**Stale seat count persists when navigating to a non-multi-pax search (SPA)**

When `setSearchParams` is called without a `passengers` param (URL has no `?p=`), `parseInt(undefined, 10)` returns `NaN`, `NaN > 0` is `false`, so `dispatch(tripSummaryActions.setTravelerInfo({ seats: parsedPassengers }))` is **never called**. The trip summary Redux store retains whatever `seats` value was set during the previous multi-pax search.

**Concrete scenario:** User searches for 3 passengers (`?p=3`), sees multi-pax results. They click back or navigate to a different route with no `?p=` param. `setSearchParams` is invoked, stale `seats: 3` remains in the store. `selectTripPrices` in `wanderu-ducks-trips` now multiplies every listed price by 3 (the new `multipliedPricing` path). User sees 3× the real price for a single-passenger search.

**Evidence:** Challenger confirmed. `resetTravelerInfo` exists but is a dead prop (mapped in `mapDispatchToProps` but never called from the component body). No other dispatch resets `seats` to 1 during SPA navigation. `selectTravelerInfo` initial state `seats: 1` only applies at store initialization.

**Fix:** Always dispatch `setTravelerInfo` when `setSearchParams` runs, defaulting to 1 when `passengers` is absent:
```js
const parsedPassengers = parseInt(passengers, 10);
dispatch(tripSummaryActions.setTravelerInfo({ seats: parsedPassengers > 0 ? parsedPassengers : 1 }));
```
Same pattern needed in `TripSummary.tsx:325–330` for the SSR path.

### SUGGESTIONS (2)

#### [2] `packages/wanderu-ducks-trips/src/appstate/selectors.js:311–321`
**No tests for the `multipliedPricing` path — the core of the feature**

The new `multipliedPricing` branch (reached when `seats > 1` and no live-check pricing) has zero test coverage. `selectTripPrices` is not referenced in any test in `packages/wanderu-ducks-trips/src/appstate/tests/`. This is the pricing logic users will see for every multi-pax trip where live-check hasn't run yet — a wrong multiplication would silently show incorrect prices to the cohort under the flag.

**Evidence:** `grep -r "selectTripPrices" packages/wanderu-ducks-trips/src/appstate/tests/` returns no matches.

**Specific missing cases:**
- `seats=2`, no LC pricing: each currency value should double
- Multi-leg trip with `seats=3` and no LC: each leg multiplied independently, then summed
- `lcPricing` present with `seats > 1`: `_perSeatPricing` uses `basePricing` (per-seat) while `pricing` uses `lcPricing` (already-totaled) — no test verifies this asymmetry

#### [3] `packages/wanderu-widget/src/components/SearchWidget/PassengerSelect/` — new directory
**No tests for the new `PassengerSelect` component**

`PassengerSelect.jsx` has no test file. The component has non-trivial behavior: `handleSelectionChange` coerces the key to string before calling `onChange`, the default value is `'1'`, the plural label logic uses `value > 1`. Per CLAUDE.md: TDD is non-negotiable.

**Most important missing case:** A test asserting `value="2"` renders "2 Passengers" (plural) would also surface the string/number coercion discussed in DISPUTED finding below.

### DISPUTED (1 — not posted unless you approve)

*Passed verified-analysis but an independent challenger found mitigating evidence.*

#### [D1] `packages/wanderu-widget/src/components/SearchWidget/PassengerSelect/PassengerSelect.jsx:40`
**`value > 1` compares string to number implicitly**

`value` is `PropTypes.string` and `PASSENGER_OPTIONS` generates strings. `'2' > 1` works by JS coercion but is type-inconsistent. Fix: `Number(value) > 1`.

**Challenger: Weakened** — all entry points produce clean numeric strings. A non-numeric prop would be API misuse. The pattern is consistent with how `passengersInput` is typed throughout.
