### SUGGESTIONS

**S1 — `toError()` not used in catch blocks (`src/observability/client.ts:33, 94`)**

Both catch blocks use `error instanceof Error ? error : new Error(String(error))` instead of `toError()` from `@/lib/errors`. CLAUDE.md is explicit about using `toError()`. The two are not equivalent: `toError()` extracts `.message` from non-Error objects and falls back to `JSON.stringify` before `String()`. Line 94 is inside the `if (rumEnabled)` block this PR significantly modified.

Fix:
```ts
import { toError } from '@/lib/errors';
.withError(toError(error)).error('Failed to initialize RUM');
```

**S2 — `match` function (lines 50–62) has zero test coverage**

The `match` function — which decides whether to inject trace headers into outgoing requests — has no tests. All 3 branches uncovered (CI confirmed: lines 50–62). The environment supports tests: file is already `@vitest-environment jsdom`, `window.location` is available. Extract `match` from `vi.mocked(datadogRum.init).mock.calls[0][0].allowedTracingUrls[0].match` and test: same-origin `/api/route`, cross-origin URL, `__nextjs` URL, malformed string.

### NITS

**N1 — `match` comment omits that Next.js internal paths are also same-origin (`client.ts:43–48`)**

Comment says "same-origin only" but `__nextjs`/webpack paths are same-origin — the carve-out is unexplained. Add a line noting these don't hit the OTel-instrumented server.

## ui-react #868 — MAIN-5173 Multi-pax

**URL:** https://github.com/wanderu/ui-react/pull/868  
**Author:** noahwelch  
**Classification:** NEEDS_REVIEW  
**Review action:** REQUEST_CHANGES (1 blocking, 3 suggestions, 2 nits)

### Context

Adds a 1–8 passenger dropdown to the search widget behind `search-multi-pax` feature flag. 35 files, 617 additions, 205 deletions. Passengers flow as string `?p=` URL param through `wanderu-ducks-widget` and `wanderu-ducks-search` Redux slices, then parsed to a number for livecheck API and trip summary pricing. Client-side price multiplication (`base × seats`) added to `selectTripPrices` when no live-check pricing available.

### BLOCKING

**B1 — URL passenger count (`?p=3`) silently ignored; widget always shows 1 passenger**
`packages/wanderu-widget/src/components/SearchWidget/SearchWidget.jsx:288–292`

In `componentDidUpdate`, the condition `if (passengersInput && reduxPassengersInput && passengersInput !== reduxPassengersInput)` requires `reduxPassengersInput` to be truthy. `wanderu-ducks-widget` reducer initializes `passengersInput: undefined`. On initial load with `?p=3` in the URL, `reduxPassengersInput=undefined` and the `&&` short-circuits — Redux is never updated. Confirmed: `searchUrlManager`'s `setSearchParams` dispatch only updates `wanderu-ducks-search` slices, not `wanderu-ducks-widget`. `PassengerSelect` renders `value={passengersInput}` which stays `undefined`, defaulting to 1 passenger regardless of `?p=`.

Fix: Remove the `reduxPassengersInput` truthy guard:
```js
if (passengersInput && passengersInput !== reduxPassengersInput) {
    setPassengersInput(passengersInput);
}
```
